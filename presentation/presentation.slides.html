Mean time to remediation is dominated by lookup time, not fix time

~30s. Three pain points every ops team recognises. The fix isn't faster engineers — it's institutional memory made queryable. That's what this pipeline does.
Architecture

How It Works

[01] INCIDENT ServiceNow ticket free-text description
→
[02] RAG SEARCH sentence-transformers + ChromaDB top-3 matches
→
[03] LLM REASONING Ollama qwen3:14b → structured JSON response
→
[04] DECISION GATE ≥0.70 auto-fire · <0.70 human review
→
[05] OUTCOME Rundeck job fires or engineer gets pre-reasoned brief
▸ Fully local  ·  No data leaves the machine  ·  No API costs  ·  ~10s end-to-end
~45s. Walk through each stage left to right. Emphasise: the LLM never sees raw data — it only gets the RAG context already pulled from the knowledge base. Everything runs on hardware you own.

~30s. Two inputs to the RAG: incident history and runbooks. Both are embedded the same way. The system sees them both as equal-weight context chunks — a runbook match is treated the same as an incident match.

Roadmap

Where This Goes Next

Shadow Deployment Pipeline

A parallel shadow environment mirrors all live traffic and incoming incidents without taking any action. When the pipeline's recommendations have earned sufficient confidence, a single approval promotes the shadow's configuration to production and tears down the shadow instance. Zero-downtime, zero-guesswork rollout — the system validates itself before it acts.

Repository-Aware Context

Continuous indexing of service source repositories gives the LLM access to code-level context — configuration files, dependency changes, recent commits — when an incident's root cause demands investigation beyond historical tickets and runbooks. The knowledge base grows to match the depth of the problem.

Living Organisational Memory

Automatic ingestion of JIRA tickets, Confluence pages, and equivalent knowledge bases keeps the RAG context current with the team's accumulated decisions, post-mortems, and architectural changes. The system learns continuously from every resolved incident and every documented decision — whether or not it was the one that triaged it.

~30s. Three natural extensions. Shadow deployment closes the gap between recommendation and action. Repository indexing solves the "I need to look at the code" problem. Living organisational memory means the system never forgets a decision made in Confluence or a lesson captured in a JIRA post-mortem.

Production Readiness

Observability

Log Every Step

RAG top-K + similarity scores, LLM token counts + latency, every gate confidence and outcome, every Rundeck execution result.

Dashboards

Confidence distribution over time, auto-execution rate, false positive rate, and P99 latency per pipeline stage.

Alerts

Confidence distribution shift, fallback rate increase, and any Rundeck failure without a subsequent human resolution.

Sudden confidence drops are a production smell: stale knowledge base, model version drift, or a retrieval-quality regression.

~30s. The observability story is straightforward: log the retrieval, model call, gate decision, and execution outcome. From that, build confidence and latency dashboards and alert on drift.
