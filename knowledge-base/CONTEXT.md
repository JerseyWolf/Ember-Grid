# knowledge-base — Ember Grid operational knowledge corpus

This directory is the inner ring of the three-ring architecture: the
ground-truth corpus that everything else in `ops-knowledge-loop` reads
from. Six handwritten runbooks plus two systems-level documents are
chunked, embedded and served via the `rag/` ring.

## Routing Table

| Task | Read | Skip | Notes |
|------|------|------|-------|
| Understand a service's failure modes | `runbooks/<service-name>.md` | systems/ | One file per service; consistent structure. |
| Map a service to its team and on-call | `systems/service-map.md` | runbooks/ | The single source of truth for ownership. |
| Understand the platform topology | `systems/ember-grid-architecture.md` | runbooks/ | High-level only; not for incident triage. |
| Find an auto-generated runbook from a past incident | `incidents/INC*.md` | runbooks/ | Generated by `incident_pipeline/generate_runbook.py`. |
| Add a new runbook | `runbooks/checkout-service.md` (as template) | — | Then run `python rag/populate_database.py`. |

## Entry Point

This directory is read by the RAG ingest script:

    python rag/populate_database.py

After ingest, semantic search becomes available via `rag/query.py`.

## Inputs

- Handwritten markdown by on-call engineers (six runbooks today).
- Auto-generated incident reports written to `incidents/` by
  `incident_pipeline/generate_runbook.py`.
- Two systems-level documents under `systems/`.

## Outputs

- Nothing is produced from this directory directly. The downstream effect
  is that `chroma_db/` becomes populated and `rag/query.py` returns
  relevant chunks.

## Demo Talking Point

"Every markdown file in here goes through the same RAG ingest, so the
auto-generated runbook from an incident at 02:00 is searchable by the
on-call engineer at 02:05 — that is the entire loop we are building."
