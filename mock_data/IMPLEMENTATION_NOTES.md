# Implementation Notes — mock_data

## What Was Built

Three JSON fixtures: `incidents.json` (100 realistic Ember Grid incidents,
pre-generated, never overwritten); `rundeck_jobs.json` (10 Rundeck jobs
covering every remediation referenced in the runbooks, with risk
levels and per-job confidence thresholds); `servicenow_responses.json`
(a sample ServiceNow REST envelope with 3 open P2 incidents matching
the real API shape).

## Key Design Decisions

- Decision: Treat `incidents.json` as a hard pre-existing input —
  Reason: the file was generated by a separate process and the rest of
  the pipeline depends on its exact content. Treat it as immutable demo
  input unless deliberately regenerating the knowledge base.
- Decision: Per-job `confidence_threshold` and `risk_level` in
  `rundeck_jobs.json` — Reason: the confidence gate in
  `incident_pipeline/trigger_rundeck.py` reads them as data, not code.
  This keeps the policy-vs-mechanism boundary clean.
- Decision: `human_review_required: true` flag on
  `trigger-manual-fulfilment-retry` — Reason: the order-fulfilment
  duplicate-shipment risk should never auto-execute, regardless of
  confidence score. Encoding it as data is safer than burying it in code.
- Decision: `servicenow_responses.json` mirrors the real ServiceNow
  table API envelope (`result[]`, `sys_id`, `state` as numeric string,
  `assigned_to` as nested dict) — Reason: anyone reading the demo can
  see exactly what the real integration would receive.

## How It Fits the Architecture

The outer ring's contract surface. `incident_pipeline/fetch_incidents.py`
reads `incidents.json` in MOCK_MODE; `incident_pipeline/ai_remediation.py`
and `trigger_rundeck.py` read `rundeck_jobs.json`. Switching to real
mode changes the source but not the shape — that is the point.

## How to Extend

- To add a new data source: drop a new JSON fixture into this
  directory and add a reader to the relevant pipeline module. Keep
  the file's shape identical to its real-mode counterpart.
- To swap the LLM: this directory has nothing to change.

## Demo Talking Points

- "Every file in here matches the shape of the real upstream. Flipping
  `MOCK_MODE=false` swaps the source, not the contract."
- "The Rundeck catalogue has per-job risk and confidence metadata. The
  gate that protects production is enforced from this JSON, not from
  hidden logic in the pipeline code."

## Known Limitations (Honest)

- No schema validation. A JSON Schema for each file would make the
  contract enforceable at PR time via OPA or a small pytest. Worth
  doing in a follow-up.
- `incidents.json` is fixed at 100 records. For larger-scale testing
  the fixture would need to be regenerated; `incidents.json` is read-
  only by policy so the regenerator would write to a new file.
